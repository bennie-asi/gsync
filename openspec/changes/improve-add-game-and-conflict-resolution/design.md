## Context

GSYNC 当前已经有可运行的 Add Game 向导、游戏详情页、冲突解决页和同步引擎，但这几块仍处于“安全基础版”状态：向导中的游戏匹配主要依赖简单显示名匹配，路径复查更多是展示而非约束；冲突解决页可以表达差异，但实际应用动作仍以整组上传/下载为主。现有 UI 已经按照 Stitch refined 结构落地，因此本次设计的核心约束不是重新设计页面，而是在不破坏既有骨架的前提下增强行为质量和安全级别。

这次变更同时横跨 UI、应用层和同步引擎：Add Game 向导涉及扫描结果归并、清单匹配、路径预检、用户定义落盘；冲突解决涉及比较结果建模、文件级决策表达以及选择性应用。由于这些决策会影响后续实现边界和数据流，所以需要单独设计说明。

## Goals / Non-Goals

**Goals:**
- 为 Add Game 向导引入可解释的匹配流程，而不是只做简单名称命中。
- 在创建配置前执行更明确的路径预检，让用户能区分“可创建但需要注意”和“明显无效不应继续”的情况。
- 为冲突解决引入文件级决策模型，允许同一组冲突中混合保留本地/远端文件。
- 在同步引擎中实现选择性应用路径，并保持下载前快照保护语义不变。
- 保持现有 Stitch refined UI 骨架和高密度桌面工具风格。

**Non-Goals:**
- 不在这次变更中重做整套导航、页面布局或视觉设计。
- 不引入新的外部存储 provider 或新的持久化引擎。
- 不在这次变更中实现自动冲突合并、三方文本合并或二进制差异工具。
- 不解决所有社区清单匹配问题；目标是把匹配质量从“仅简单命中”提升到“可解释且可回退”。

## Decisions

### 1. Add Game 匹配改为“分层候选 + 明确回退”
- **Decision**: 为扫描结果与内容定义之间建立分层匹配规则，优先使用稳定标识（如 source hints / platform id），其次使用规范化名称匹配，最后回退到手动配置。
- **Why**: 仅靠显示名会让同名、别名、带版本后缀或平台命名差异的游戏频繁误判。分层候选可以提升命中质量，同时保留可解释性。
- **Alternatives considered**:
  - 继续使用显示名完全匹配：实现简单，但误判率高。
  - 引入复杂模糊匹配库：命中率可能更高，但对于当前产品阶段过重，而且可解释性较差。

### 2. 路径预检分为“阻断错误”和“非阻断警告”
- **Decision**: 在向导 Review Paths 步骤中，对每个启用内容项输出预检状态，并区分阻断错误（例如所有启用内容路径都无法解析）与非阻断警告（路径当前不存在、依赖用户后续启动游戏生成等）。
- **Why**: 现实中很多游戏目录在首次运行前并不会生成完整存档路径，如果一律阻断会影响可用性；但若完全不区分，又会让用户创建出明显错误的配置。
- **Alternatives considered**:
  - 所有不存在路径一律阻断：更安全，但会误伤大量合理场景。
  - 所有问题都仅作为提示：更宽松，但容易让无效配置进入系统。

### 3. 手动配置继续落在用户定义内容模型中
- **Decision**: 未匹配或手动录入的游戏继续通过用户定义内容文件持久化，而不是把临时定义只留在数据库中。
- **Why**: 这样能与现有 manifest/service 流程保持一致，也便于后续调试、导出和演进。
- **Alternatives considered**:
  - 仅把手动定义写入数据库：实现快，但会引入第二套内容定义来源。
  - 强制所有游戏必须命中社区定义：产品体验过于受限。

### 4. 冲突解决采用“文件级决策清单”而不是一次性全局策略
- **Decision**: 在比较结果上增加可编辑的决策层，每个冲突文件都可以标记为保留本地、保留远端、暂不处理；页面再提供批量快捷动作来批量设置这些决策。
- **Why**: 用户常常只想保留局部本地改动，而不是对整组内容统一上传或统一下载。文件级决策更符合实际冲突处理心智模型。
- **Alternatives considered**:
  - 仅保留“全部本地 / 全部远端”：实现最简单，但过于粗粒度。
  - 引入逐字节合并工具：超出当前产品阶段和风险承受范围。

### 5. 选择性应用通过扩展 SyncEngine 执行计划完成
- **Decision**: 在 SyncEngine 中新增“基于冲突决策的执行计划”路径，由应用层把文件级决策映射成受控的上传/下载/跳过操作；任何会覆盖本地文件的执行计划仍复用下载前快照保护。
- **Why**: 把选择性应用收敛到 SyncEngine 能保持同步语义集中，避免在 UI 层直接拼装文件覆盖逻辑。
- **Alternatives considered**:
  - 在 UI 层直接读写文件：会绕开同步引擎的安全边界。
  - 为每个文件生成独立 SyncJob：更贴合现有结构，但会让队列和历史噪声过大。

## Risks / Trade-offs

- **[匹配规则仍可能命中错误定义]** → 通过显示匹配依据、允许用户手动回退，以及默认在 Finish 步骤再次展示内容定义摘要来降低误建风险。
- **[路径预检过严或过松都会伤害体验]** → 采用阻断错误/非阻断警告分层，并把判定逻辑集中在单独服务中，方便后续调整规则。
- **[文件级冲突决策会增加 UI 复杂度]** → 保持默认批量动作入口，同时只在需要时展开逐文件选择，避免基础用户被迫理解全部细节。
- **[选择性应用会让 SyncEngine 逻辑变复杂]** → 通过显式执行计划对象和集中单元测试覆盖，把复杂度收敛在可测试的核心层。
- **[手动定义文件可能与未来社区定义重复]** → 后续可增加“用社区定义替换手动定义”的迁移路径；本次先确保行为正确和可追踪。
