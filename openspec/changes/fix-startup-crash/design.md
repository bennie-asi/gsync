## Context

当前仓库的实现方向已经明确为 WinUI 3 桌面应用，并且 UI 第二阶段正在按 Stitch refined / normalized 设计稿推进。近期排查表明，启动闪退并非由 SQLite 数据文件损坏或 `DatabaseInitializer` 的基础逻辑直接导致：独立数据库探针成功、最小化 `App.xaml` + `MainWindow` 可以启动、恢复共享资源链与基础壳层控件后仍可启动，说明问题更集中在完整启动链路重新接入后的页面初始化、导航入口或启动时机控制上。

现有已验证事实还显示：`HistoryPage`、`SyncTargetsPage`、`VariablesPage` 在单独作为启动页时可以成功打开；`SettingsPage` 与 `HomePage/Library` 仍是高优先级怀疑对象。项目当前最需要的不是继续扩大怀疑面，而是把启动过程拆分成安全阶段，让主窗口壳层先稳定呈现，再以受控方式恢复页面加载与运行时逻辑，并在失败时留下可诊断信号而不是直接崩溃退出。

该变更同时受到以下约束：
- 必须继续遵循 Stitch 设计稿，不得因为修复闪退而重设计界面结构。
- 启动修复应优先稳定主壳层与首屏呈现，不把问题继续归咎于已经基本排除的数据库基线。
- 页面级异常应被隔离在页面或导航层，不得升级为应用整体退出。

## Goals / Non-Goals

**Goals:**
- 让应用在冷启动与常规启动时稳定显示主窗口壳层。
- 将启动过程拆分为“应用资源初始化 → 主窗口壳层呈现 → 默认页面导航 → 页面异步加载”几个可观测阶段。
- 为首屏 Library 以及管理页面建立统一的异常隔离策略，避免页面初始化错误直接造成闪退。
- 为启动和导航失败补充结构化日志与可见的降级结果，便于后续继续定位根因。
- 为后续恢复完整首页与 Settings 页面逻辑提供稳定基线，而不是长期停留在最小探针状态。

**Non-Goals:**
- 不在本次设计中重做整套 UI 信息架构或偏离 Stitch 设计稿。
- 不重构数据库、同步引擎或完整数据层架构。
- 不一次性解决所有页面内部功能缺失；本次目标是先保证“能稳定启动并可继续诊断”。
- 不引入新的外部依赖或全新的应用框架。

## Decisions

### Decision: 启动采用分阶段恢复，而不是一次性恢复完整逻辑
**Decision**
启动流程将被明确拆分为几个阶段：
1. `App` 级资源与基础服务初始化
2. `MainWindow` 壳层创建并显示
3. 壳层中的默认导航目标解析
4. 目标页面自身的数据加载与运行时逻辑触发

只有前一阶段成功后，下一阶段才继续；页面数据加载应尽量从同步构造/初始化迁移到可捕获失败的异步入口。

**Rationale**
当前排查结果已经表明，最小壳层是安全的，问题出现在“恢复完整启动链”之后。与其继续做大范围猜测，不如把启动链拆成可验证阶段，快速缩小具体失败位置。

**Alternatives considered**
- **一次性恢复完整旧逻辑并继续试错**：实现快，但每次失败仍然表现为整体闪退，定位效率低。
- **长期停留在最小壳层探针**：可以回避崩溃，但无法恢复真实产品路径，不能支撑后续 UI 验证。

### Decision: 主窗口壳层优先显示，页面失败只影响内容区
**Decision**
`MainWindow` 壳层（标题栏、导航栏、Frame/内容承载区、状态栏）应在默认页面成功完成全部业务加载之前就可以显示。页面初始化失败时，应用保留壳层与导航可用，内容区展示可诊断的错误或安全占位状态，而不是使窗口创建失败或应用退出。

**Rationale**
现有验证显示壳层本身不是主要问题。壳层先行稳定呈现，可以把故障从“应用无法启动”降级为“某页面无法正常初始化”，同时保持用户和开发者可见性。

**Alternatives considered**
- **窗口显示依赖首页完整初始化成功**：会把页面错误放大为启动闪退。
- **发现异常后静默吞掉并显示空白页**：虽然不闪退，但缺乏可诊断性，后续问题更难定位。

### Decision: 页面初始化逻辑与导航逻辑解耦
**Decision**
默认页面选择、导航注册和页面内部数据加载需要分开处理。导航层只负责将用户带到目标页面；页面自身的 ViewModel 加载、命令预热、样本数据绑定、外部服务访问等逻辑应放在可控入口中，并允许失败后展示页面内错误状态。

**Rationale**
从现有迹象看，Library 和 Settings 更可能在页面自身初始化阶段触发异常。把导航与页面数据加载拆开，可以验证究竟是“无法导航”还是“导航后页面逻辑失败”。

**Alternatives considered**
- **继续在页面构造函数或同步 Loaded 事件中堆叠初始化逻辑**：实现简单，但异常边界差。
- **把所有逻辑都上提到 App 启动阶段**：会加重 App 入口复杂度，也让启动失败面扩大。

### Decision: 启动与导航失败必须留下结构化诊断信息
**Decision**
对以下关键点记录结构化日志：应用启动入口、主窗口创建、默认导航目标、页面导航完成、页面初始化开始/成功/失败。若存在未处理异常入口，也应在捕获后写入日志并尽量保留用户可见壳层或错误窗口。

**Rationale**
当前排查能取得进展，依赖的就是逐步排除法。若没有稳定、细粒度的诊断信号，后续每次恢复逻辑都可能回到黑盒式试错。

**Alternatives considered**
- **仅依赖调试器和手工复现**：适合一次性排查，不适合持续恢复完整启动链。
- **只记录最终异常堆栈**：缺少阶段上下文，仍难判断失败发生在何处。

### Decision: 首屏与管理页面统一采用“页面内错误状态”策略
**Decision**
Library、Settings、History、Sync Targets、Variables 等页面在初始化异常时，优先保留页面外层布局与 Stitch 壳层一致性，仅在内容区显示页面级错误/降级状态，并允许用户切换到其他页面继续使用应用。

**Rationale**
这与现有管理页面状态变体思路一致：保持壳层与布局稳定，只替换焦点区域。这样既符合桌面工具风格，也能避免页面异常演变成全局不可用。

**Alternatives considered**
- **异常时弹全局阻塞对话框并退出**：对用户破坏性太强。
- **每个页面各自定义完全不同的错误处理方式**：会造成行为不一致，增加维护成本。

## Risks / Trade-offs

- **[风险] 分阶段恢复可能暂时暴露更多“页面内错误状态”** → **Mitigation:** 接受短期内从“闪退”降级为“页面异常可见”，优先恢复可启动基线，再逐页修复。
- **[风险] 将初始化改为异步后，首屏可能出现短暂加载/占位状态** → **Mitigation:** 使用与 Stitch 风格一致的紧凑加载或错误占位，不改变页面整体布局。
- **[风险] 过度捕获异常可能掩盖真正的崩溃根因** → **Mitigation:** 只在壳层/导航边界进行捕获并强制写日志，不在深层逻辑中无差别吞异常。
- **[风险] 启动日志增多导致噪音** → **Mitigation:** 只记录关键生命周期节点，避免把所有 UI 细节都写入启动日志。

## Migration Plan

1. 以当前已验证的安全壳层为基线，确认 `App.xaml`、`MainWindow.xaml`、`MainWindow.xaml.cs` 的最小稳定组合。
2. 明确默认导航入口与页面初始化入口，先让壳层显示，再恢复默认页导航。
3. 优先对 Library 和 Settings 应用页面级异常隔离与日志记录，因为它们是当前最高优先级怀疑点。
4. 将同样的导航/初始化边界策略推广到其他管理页面，统一故障行为。
5. 通过本地运行验证冷启动、正常启动和页面切换路径，确保失败时应用不再直接退出。
6. 若恢复过程中某一页面再次导致整体闪退，可回退到上一安全阶段并保留新增日志，避免重新回到无诊断状态。

## Open Questions

- Library 当前的具体崩溃触发点是在页面构造、ViewModel 绑定、导航后事件，还是首批数据加载命令中？
- SettingsPage 是否与资源字典、样式引用、绑定错误或特定控件初始化有关？
- 当前项目中是否已经存在统一的日志/异常上报基础设施可复用，还是需要先补最小化生命周期日志？
- 启动失败时的最佳可视化降级形式应为页面内错误面板、导航可切换的空状态，还是独立安全页？
