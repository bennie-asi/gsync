## Why

当前项目在启动阶段存在闪退问题，导致用户无法稳定进入主界面并阻断后续 UI 与功能验证。现有排查已经基本排除数据库文件损坏与基础初始化链路本身，更需要通过一次明确的变更，把启动流程、页面初始化边界和故障诊断能力收敛成可落实的修复范围。

## What Changes

- 修复应用在冷启动和正常启动路径中的闪退问题，确保用户能够稳定进入主界面壳层。
- 收敛启动阶段的职责边界，避免高风险页面或运行时逻辑在窗口创建时同步阻塞或直接触发崩溃。
- 为启动阶段补充可定位的诊断信息与安全降级路径，使页面初始化异常不会直接表现为整体应用闪退。
- 明确 Library、Settings 及其他首屏/导航目标页面的初始化约束，保证现有 Stitch 对齐的 UI 壳层可以先稳定呈现，再逐步恢复页面逻辑。

## Capabilities

### New Capabilities
- `app-startup-stability`: 定义应用启动、主窗口呈现、首屏导航初始化、异常隔离与启动诊断的稳定性要求。

### Modified Capabilities
- `stitch-app-shell`: 调整统一应用壳相关要求，使主窗口壳层在启动时优先稳定呈现，并与受控的页面加载顺序配合。
- `library-and-details-ui`: 调整 Library 首屏相关要求，明确其初始化失败时不得导致整个应用在启动阶段退出。
- `management-screens-ui`: 调整 Settings、History、Sync Targets、Variables 等管理页面相关要求，明确其页面初始化异常不得在启动或导航阶段直接触发整体崩溃。

## Impact

- 受影响范围包括应用启动入口、主窗口壳层初始化、首屏导航与页面加载顺序。
- 受影响系统包括异常处理、日志/诊断记录、页面级 ViewModel 或数据加载触发时机。
- 该变更不引入新的外部产品能力，但会改变启动阶段的实现约束与页面初始化策略。
- 修复完成后，可为后续继续按 Stitch 设计稿推进 UI 实现提供稳定基线。
