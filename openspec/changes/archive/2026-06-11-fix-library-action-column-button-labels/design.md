## Context

用户反馈 Library 主表格的操作列按钮文字已经看不见或被裁切，这说明问题已经从“视觉细节偏差”升级为“直接影响主操作可用性”的缺陷。当前 `HomePage.xaml` 在 `LibraryActionCellTemplate` 中把两个按钮并排塞进操作列，而 `HomePage.xaml.cs` 给操作列只分配了 `Width = 118, MinWidth = 110`，在当前按钮默认 Padding 下，这个宽度不足以同时容纳“打开 / 更多”两枚文本按钮，因此文字被严重压缩甚至不可见。

这次修复应当保持范围极小：优先调整操作列模板与列宽，让按钮文字重新可见，同时尽量不破坏 Library 表格的整体密度。由于 `ResizableTableView` 已经具备列宽控制能力，而问题目前集中在单个列模板和初始列宽配置上，因此这次更适合做页面级精准修复，而不是扩展整套表格 primitive 行为。

## Goals / Non-Goals

**Goals:**
- 修复 Library 主表格操作列中按钮文字不可见或被裁切的问题。
- 让“打开 / 更多”按钮在当前表格密度下仍然清晰可辨并可点击。
- 保持 Library 表格整体结构和 Stitch refined 风格不被明显破坏。
- 将改动聚焦在操作列模板、按钮布局和列宽配置上。

**Non-Goals:**
- 不重做整个 Library 主表格或 `ResizableTableView` 的通用布局体系。
- 不重构 Library 页面其他列、右侧时间线区域或主壳层。
- 不新增新的表格组件或新的交互行为。
- 不在本轮扩展到其他页面的操作列问题。

## Decisions

### 决策 1：优先修正操作列模板与列宽，不修改整张表格架构
- 决定：先通过 `LibraryActionCellTemplate` 和 `HomePage.xaml.cs` 中 action 列宽配置来解决问题。
- 原因：问题来源明确，当前列宽与双按钮布局不匹配，没必要放大到 `ResizableTableView` 重构。
- 备选方案：
  - 修改 `ResizableTableView` 的全局单元格承载逻辑：影响面过大。

### 决策 2：保持双按钮，但收紧布局并给出足够列宽
- 决定：继续保留“打开 / 更多”双按钮操作，但要么减小单元格内部间距与按钮 padding，要么提高 action 列初始宽度，必要时两者一起做。
- 原因：用户反馈的是“字看不到”，不是要求减少按钮数量；优先保持原有信息架构。
- 备选方案：
  - 把其中一个按钮改成纯图标：会改变当前页面交互表达，不是最小修复。

### 决策 3：优先在 HomePage 页面局部修复，不先改全局按钮样式
- 决定：如果要调整按钮 padding，优先对 `LibraryActionCellTemplate` 局部设置，而不是直接改 `SecondaryToolbarButtonStyle`。
- 原因：全局按钮样式会影响整个应用，风险不必要地扩大。
- 备选方案：
  - 全局缩小所有 secondary button 的 padding：可能破坏其他已对齐页面。

## Risks / Trade-offs

- [单纯增大 action 列宽可能挤压相邻列] → 只做适度增宽，并结合模板内部间距收紧，避免让整表比例明显失衡。
- [单纯缩小按钮 padding 可能仍不足以显示完整文字] → 需要同时验证操作列宽和按钮最小可读宽度。
- [如果继续保留两个文本按钮，极窄窗口下仍可能紧张] → 以当前默认窗口与 refined 稿对齐为主，保证正常场景优先可用。
- [若修改全局样式会影响其他页面] → 坚持先做页面局部修复，不扩大到全局样式层。

## Migration Plan

1. 在 proposal 中锁定这是 Library 操作列文字不可见的精准修复。
2. 在 specs 中补充 `library-and-details-ui` 对操作列可读性的要求。
3. 在 tasks 中按“确认问题来源 → 修模板/列宽 → 构建验证”拆分。
4. 实施时先修改 `HomePage.xaml` 与 `HomePage.xaml.cs`，只有页面级修复不足时再考虑碰 `ResizableTableView`。
5. 最后通过构建和实际渲染验证确认按钮文字已恢复可见。

## Open Questions

- 最终修复是“保持双文本按钮 + 增宽列”还是“保持列宽基本不变 + 更紧凑的按钮布局”，当前建议优先做两者的平衡。
- 如果用户后续希望更贴近设计稿，是否需要把“更多”按钮收敛成更轻量表达，但这不属于本轮最小修复范围。
