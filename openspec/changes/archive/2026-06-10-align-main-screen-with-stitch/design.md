## Context

当前仓库的 UI 工作仍处于以规划与设计稿为主的阶段，但 OpenSpec 中已经定义了 `stitch-app-shell` 与 `library-and-details-ui` 两类 UI 约束。用户反馈当前主界面实现与 Stitch refined Library 设计稿存在明显偏差，主要集中在三类问题：整体壳层结构不一致、Library 页面主功能区布局不一致、以及状态颜色在中文本地化下失去语义区分。

这次变更需要同时约束壳层、页面分区和状态语义。它不是单个控件的小修，而是一次以 Stitch 设计稿为验收基准的结构性收口，且后续实现需要优先参考 Stitch 项目 `13407775155513183369` 中的 refined Library 相关屏幕，而不是继续沿用当前偏离版本。

## Goals / Non-Goals

**Goals:**
- 明确 Library 页面的最终信息架构与区域分布，作为后续实现与验收基准。
- 明确应用壳在 Library 页面上的固定元素，包括左侧菜单头部与底部状态栏。
- 明确状态文本的“语义→颜色”映射应独立于显示语言，避免中文翻译后丢失状态色。
- 把后续实现约束绑定到 Stitch refined 设计稿，而不是允许自由再设计。

**Non-Goals:**
- 不重新定义整个产品的导航结构或增加新的业务能力。
- 不在本次设计中扩展 Game Details、History、Variables 等非当前反馈重点页面的布局。
- 不规定具体 UI 框架控件类型、样式类名或 ViewModel 属性命名。
- 不改变现有状态语义集合，只修正其布局承载方式和多语言颜色表达规则。

## Decisions

### 决策 1：以 refined Library 设计稿作为主界面结构唯一基准
- 决定：Library 页面实现必须对齐 Stitch refined Library，而不是允许按“近似桌面风格”自由发挥。
- 原因：当前偏差主要来自实现阶段对设计稿进行了结构性再解释；如果继续只保留高层原则，后续页面仍会持续偏离。
- 备选方案：
  - 仅用文字规范约束桌面工具风格：约束太弱，无法避免区域布局继续偏移。
  - 允许实现做适度再设计：会再次引入“看起来差不多但结构不一致”的问题。

### 决策 2：Library 页面采用“三段式主区 + 固定底栏”布局
- 决定：页面主体按左侧固定菜单、中央表格主区、右侧操作/总览区组织；底部状态栏固定显示，不随内容滚动消失。
- 原因：这与用户从设计稿读出的信息架构一致，也符合桌面工具对常驻状态信息的预期。
- 备选方案：
  - 将底部状态栏并入主内容卡片：会弱化运行状态的常驻可见性。
  - 将右侧操作和总览合并到顶部工具栏：会压缩主表格空间，且与设计稿不一致。

### 决策 3：搜索、排序、刷新与主操作分区独立
- 决定：搜索输入框放在主功能区顶部并占据较大横向权重；排序/刷新作为同一行右侧次级操作；“添加游戏”和“立即同步”放到右上操作区，避免与搜索工具混排。
- 原因：这能保持信息扫描路径清晰：先筛选列表，再查看列表，再执行右侧主操作与总览判断。
- 备选方案：
  - 把所有按钮放到同一条顶栏：会让页面顶部拥挤，且弱化右侧总览区的独立性。
  - 把搜索框缩短为普通输入框：会破坏设计稿中明显的搜索入口层级。

### 决策 4：状态颜色以语义键驱动，而不是以显示文本驱动
- 决定：状态 badge / 文本颜色必须绑定到稳定的语义键（如 synced、conflict、pending、ready、error），翻译层只负责显示文案，不参与颜色判定。
- 原因：用户反馈英文状态有彩色、中文只有单色，说明当前实现大概率把显示文案直接参与了颜色映射；这会在本地化后失效。
- 备选方案：
  - 为每种语言单独维护一套颜色映射：维护成本高且容易漏项。
  - 直接对中文翻译文本做额外判断：脆弱且不可扩展。

### 决策 5：底部状态栏采用左右两组稳定信息位
- 决定：左侧固定展示连接/本地可用性（如 WebDAV: Online、Local: Ready），右侧固定展示当前远端名称和同步状态。
- 原因：这与用户提供的设计稿阅读结果一致，并能把“环境状态”和“当前绑定目标状态”分开呈现。
- 备选方案：
  - 仅显示一个总状态摘要：信息密度不足，丢失连接与目标名称两个维度。
  - 把这些信息移入右侧总览卡：不符合固定底栏的设计意图。

## Risks / Trade-offs

- [设计稿细节在代码层未完全可见] → 以 OpenSpec 先锁定信息架构、固定区域与状态语义，后续实现时再逐项对照 Stitch 屏幕微调尺寸与间距。
- [当前实现可能已将状态颜色绑定到翻译文案] → 在实现阶段优先抽离语义状态枚举/键，再让本地化资源只负责标签文本。
- [Library 壳层要求变严后可能暴露其他页面壳不一致] → 本次先把约束收敛到 Library 必需部分，后续若其他页面暴露同类问题，再继续补充 delta spec。
- [右侧区域与主表格宽度分配在不同窗口尺寸下可能冲突] → 规范中只锁定区域职责与相对层级，不在设计文档中写死像素值，由实现阶段根据桌面最小宽度做响应式约束。

## Migration Plan

1. 先更新 OpenSpec，明确壳层、Library 布局与状态语义的目标要求。
2. 后续实现时优先调整共享壳层与状态 badge 语义映射，再调整 Library 页面布局，避免页面修完后又因共享组件返工。
3. 逐项对照 Stitch refined Library 屏幕验收：左侧菜单头部、顶部工具区、主表格、右侧操作区、总览区、底部状态栏、状态颜色。
4. 若实现后需要回滚，可分别回滚到旧壳层布局与旧状态 badge 映射，但会重新引入与 Stitch 不一致的问题，因此仅作为短期应急措施。

## Open Questions

- 右侧总览区在极窄窗口下是否允许折叠为抽屉或压缩卡片，需要在具体实现阶段结合最小窗口尺寸确认。
- 底部状态栏中的同步状态是否需要区分“idle / syncing / error / last synced”多个粒度，还是只保留设计稿中的摘要级状态。
- 状态颜色是否只用于表格状态列，还是也同步应用到右侧总览与底部状态栏中的对应状态文本。
