## Context

当前方案 A 已完成 Library、Game Details、History 和 Sync Targets 的一轮 Stitch 对稿，Variables 页成为剩余管理类页面中最直接连接产品核心模型的一页。它不仅是一个普通列表页，还承载路径变量层级、模板解析、可移植性验证与用户修正入口，因此必须同时兼顾高信息密度管理体验与诊断/测试能力。

仓库现有 `management-screens-ui` 已经把 Variables 归类到高密度管理页，但尚未把 refined Variables 页面及 parse error variant 细化成可实施的结构要求。此次变更需要把 Variables 正常态与 parse error state 一并规范化，确保实现时既能复用现有 split pane / property sheet 原语，又能把变量解析错误清晰地呈现在与正常态一致的壳层结构里。

## Goals / Non-Goals

**Goals:**
- 明确 Variables 页面对齐 Stitch refined 屏幕的最终结构，作为实现和验收基准。
- 明确 Variables 页面的顶部工具区、左侧变量列表/表格、右侧变量详情与路径模板测试区布局。
- 明确模板解析错误状态如何在保留管理页壳层和 split pane 主体的情况下呈现。
- 让 Variables 与已完成 Library / History / Sync Targets 页面在密度、边距、属性编辑区和诊断表达上保持一致。

**Non-Goals:**
- 不定义新的变量解析算法或路径模板语言。
- 不改变变量优先级、系统变量集合或 source/game instance/user override 的架构规则。
- 不在本次设计中扩展 Settings 页面或 Wizard Review Paths 页面的详细布局。
- 不要求本次设计实现真实解析引擎，只约束 UI 结构、状态语义和错误态展示。

## Decisions

### 决策 1：Variables 继续采用管理页 split pane
- 决定：页面采用左侧变量列表/表格，右侧详情与测试面板的 split pane 结构。
- 原因：Variables 既需要高密度浏览变量集合，又需要在选中项后查看说明、来源、优先级和测试结果，split pane 最适合这种主从关系。
- 备选方案：
  - 单列长表单：不利于同时浏览和测试多个变量。
  - 纯卡片流布局：会破坏管理页统一性。

### 决策 2：顶部工具区只承载列表级操作
- 决定：顶部工具区聚焦搜索、作用域/来源筛选、刷新、添加变量等列表级操作；具体编辑和测试动作下沉到右侧面板。
- 原因：这样能保持顶部简洁，把测试/修复动作与具体变量上下文绑定。
- 备选方案：
  - 把测试模板、编辑、删除都放到页顶：会让工具区过载，削弱右侧详情面板的意义。

### 决策 3：右侧面板同时承载详情与模板测试
- 决定：右侧详情区不仅展示变量属性，还需要包含路径模板测试结果/解析摘要，形成一个“详情 + 诊断”复合面板。
- 原因：Variables 页的价值不只是查看变量值，而是验证它们对路径模板解析的实际影响。
- 备选方案：
  - 把测试功能单独做成底部区域：会降低主从结构清晰度，也不符合 refined 页面通常的右侧 inspector 用法。

### 决策 4：parse error state 保持壳层和 pane 主体不变
- 决定：当模板解析失败时，只替换受影响的主内容区域和右侧诊断部分，保留导航、顶部栏、底部状态栏与 split pane 布局。
- 原因：Variables 错误态本质上是诊断场景，不应把整个管理页改造成独立错误页。
- 备选方案：
  - 全页错误页：与管理页错误态应“保持结构一致”的原则冲突。

### 决策 5：错误态必须紧邻修复动作
- 决定：parse error state 需要明确展示失败变量/模板、错误原因、最近一次解析结果，以及紧邻的恢复动作（例如刷新解析、编辑变量、复制模板、重试测试）。
- 原因：这类错误需要用户快速定位并修复，不能只给抽象提示。
- 备选方案：
  - 仅用错误 badge 或简短文案提示：缺少足够诊断与恢复能力。

## Risks / Trade-offs

- [右侧详情区同时承载属性和模板测试可能过于拥挤] → 通过分组标题与 PropertySheet 节奏控制层级，把测试结果放在详情之下但仍保持同一 pane。
- [现有原语对“模板测试结果”支持不足] → 优先复用 PropertySheet、KeyValueList、InfoCallout 和表格原语，只在必要时扩充少量展示模板。
- [变量来源/优先级字段未来可能扩展] → 本次只锁定信息分区与展示角色，不把字段集写死为唯一实现。
- [错误态如果处理不当容易像全页异常页] → 明确要求保留 split pane 主结构，并把错误态限定在受影响的主内容和诊断区中。

## Migration Plan

1. 先通过 OpenSpec 补齐 Variables refined 页面和 parse error state 的规范。
2. 实现时先重排顶部工具区和 split pane 主体，再收敛右侧详情/测试区。
3. 之后补 parse error state，确保错误态建立在同一壳层和 pane 结构上。
4. 最后按 refined Variables 正常态与 parse error variant 两个目标屏幕逐项验收。

## Open Questions

- refined Variables 页面中右侧测试区是否需要显示“路径模板输入 + 解析结果”双区结构，还是以单一结果摘要为主。
- parse error state 是否需要区分语法错误、缺失变量和无效作用域三类错误视觉层级，当前设计先统一为解析错误诊断态。
- 顶部工具区是否需要显式暴露“新增变量”按钮，还是将其仅放在左侧列表标题区域。
